Web development today costs less than it did five years ago.

Mmm, I remember jQuery 3.2.1, jQuery UI, and all these sliders, combo boxes, that always broke on Internet Explorer. And even in those dark times, there was always a trend for optimization and simple improvements.

Plugins were very popular. They solved problems and make your website fancier. But using them have led to new challenges as well. You did like 95% of work, and then you should go from the beginning because this is not a plugin that you are looking for.

Even now, great developers struggle with date-pickers.


Thanks to cool plugins that couldn’t ideally fit into custom cases, I was able to learn JS from professionals. I opened the source code and read it. I know it would work fantastically, and the code looked very cool too. And I tried to tweak it. No, it was not auto-updated at the time. The main goal was not to break things. It might look hard back then, but it was a different requirement.

Now you should be an air fighter, even if you only water cornfields.

Imagine a custom "Upload File" button; in some cases, it will work on Flash (OK, Boomer), and after that, you'll need to duplicate this file on different sizes and upload on S3.



 
 

 

Later, Twitter Bootstrap was born, and it was a good time for me. As a developer, I understood that with current preparations, all layouts would look amazing. But everyone was arguing with me—especially 'PSD to HTML' people. They know they can create a layout that was designed using fewer lines of code. But they always forget that design !=  a real project. And with Bootstrap and Sass, I was able to do it without their work at all :)

By using a great battle-testing modular structure, which I understand pretty well, I have solved a lot of problems in my projects. I don't need to run to the next room and beg my beloved frontend developer for help when she has some spare time.

I think that creating something from scratch and on-the-go is just a black hole for money and energy. Yeah, there are a lot of such situations.

 

But more often, playing without a proper plan will result in wasted money for project stakeholders.

 

Each project can be split into parts, then into modules, and then into 'atoms'.  And your project is huge and complex (usually). It means working under high pressure. So why are you trying to make things worse and generate more work for yourself?

 

Split it into parts and work separately on functionality, it will help you a lot.

 

Especially if your project IS VERY AGILE.

The same goes for big payment systems with proper funding using blocks/templates and default page layouts. Because when you are simplifying everything, it's hard to break it later.

 

Let's think about this:

  • you can break the wooden branch by hands;
  • you can even try to destroy it and make it very small pieces of wood;
  • you can use a knife and separate wood into cellulose fiber
But you can't break molecular bounds, do you?

Or even split an atom? This is why it rocks. So trust in physics and do your frontend work professionally.

 

Don't upset me, OK?
